# Supabase backend — setup (≈10 minutes, free tier)

This site uses **Supabase** for: contact-form submissions, user accounts/profiles,
testimonials, and blog posts. Analytics stays on Google Analytics 4 (already set up).

Everything runs on Supabase's **free tier** — no credit card needed.

## 1. Create a project

1. Go to <https://supabase.com> → **Start your project** → sign in with GitHub.
2. **New project** → pick a name (e.g. `navya-edtech`), set a database password
   (save it somewhere), choose the region closest to your users → **Create**.
3. Wait ~2 minutes for it to provision.

## 2. Create the tables

The schema is versioned as ordered migrations in [`migrations/`](./migrations).
Use the [Supabase CLI](https://supabase.com/docs/guides/cli) so every change is
tracked and applied the same way in every environment:

```bash
npm i -g supabase                       # or: brew install supabase/tap/supabase
supabase login                          # opens the browser once
supabase link --project-ref <your-ref>  # ref = the xxxx in xxxx.supabase.co
supabase db push                        # applies everything in migrations/
```

This installs the tables, indexes, triggers, RLS policies **and** the
`admin_overview_charts` aggregation function (which the dashboard uses to group
the Overview charts server-side; it falls back to client-side aggregation if
absent, so it's non-breaking).

> **No CLI?** You can still paste [`schema.sql`](./schema.sql) — a consolidated
> snapshot of all migrations — into the dashboard **SQL Editor** and run it.
> Going forward, add a new file to `migrations/` for each change rather than
> editing `schema.sql` by hand, so deployments stay ordered and reproducible.

**Making a schema change later:** add a migration and push it:

```bash
supabase migration new add_something    # creates a timestamped .sql file
# …edit the new file…
supabase db push
npm run db:types                         # refresh src/types/database.ts
```

`src/types/database.ts` is generated by `npm run db:types` (needs a linked
project). CI guards against drift: set the repo **variable**
`SUPABASE_PROJECT_REF` and the **secret** `SUPABASE_ACCESS_TOKEN`, and the
`db-types` job regenerates the types from the live schema and fails if the
committed file is stale (see `.github/workflows/ci.yml`).

(Optional) Load starter content with [`seed.sql`](./seed.sql) — the 4 starter
testimonials and one sample blog post — via the SQL Editor.

## 3. Connect the website

1. Dashboard → **Project Settings** → **API**.
2. Copy the **Project URL** and the **anon / public** key.
3. In the project root, copy `.env.example` to `.env` and fill in:

   ```env
   VITE_SUPABASE_URL=https://YOUR-PROJECT.supabase.co
   VITE_SUPABASE_ANON_KEY=eyJhbGciOi...your-anon-key...
   ```

4. Restart the dev server (`npm run dev`). The contact form now saves to the
   database, and testimonials/blog posts are read from it.

> The anon key is **meant** to be public — every row is protected by Row-Level
> Security (see the policies in `schema.sql`). Visitors can only submit contact
> messages and read *published* testimonials/posts. Never put the **service_role**
> key in the frontend.

## 4. View / manage your data

The site has a built-in **admin dashboard** at **`/admin/login`** for managing the
overview, blog posts, and testimonials. There is no public sign-up — admin accounts
are created by you:

1. Supabase dashboard → **Authentication** → **Users** → **Add user** → enter an
   email + password (tick "Auto Confirm User").
2. Promote that account to admin in the **SQL Editor**:

   ```sql
   update public.profiles set role = 'admin' where email = 'you@example.com';
   ```

3. Open the site, go to **`/admin/login`**, and sign in with those credentials.
   Non-admin accounts are rejected at login.

You can still use the dashboard's **Table Editor** to edit any row by hand.

When Supabase env vars are absent, the app falls back to bundled static data, so
local development works without any backend.

## 5. Bot protection (Cloudflare Turnstile) — optional but recommended

The contact and testimonial forms ship with a honeypot, timing checks, a
per-browser cooldown and a server-side per-email throttle. To stop a determined
bot that rotates emails, add a [Cloudflare Turnstile](https://www.cloudflare.com/products/turnstile/)
CAPTCHA verified server-side by an Edge Function:

1. **Cloudflare** → **Turnstile** → add a widget for your domain. Copy the
   **Site key** (public) and **Secret key** (private).
2. Put the site key in `.env`:
   ```env
   VITE_TURNSTILE_SITE_KEY=0x4AAAAAAA...
   ```
3. Deploy the Edge Function and give it the secret (needs the
   [Supabase CLI](https://supabase.com/docs/guides/cli)):
   ```bash
   supabase functions deploy submit --no-verify-jwt
   supabase secrets set TURNSTILE_SECRET_KEY=0x4AAAAAAA...your-secret...
   ```
   `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are injected automatically.
4. Rebuild/redeploy the site. The forms now render the challenge and submit via
   `supabase/functions/submit/index.ts`, which verifies the token before
   inserting. With no site key set, everything falls back to the direct path —
   so this is safe to roll out incrementally.

> **Full enforcement (do this once the function works):** while anonymous
> `INSERT` is still allowed by RLS, a bot could skip the widget and write to the
> tables directly. To force *all* writes through the verified function, drop the
> public insert policies so only the service-role function can write:
>
> ```sql
> drop policy if exists "contact: anyone can submit"   on public.contact_submissions;
> drop policy if exists "testimonials: public submit"  on public.testimonials;
> ```
>
> Keep these policies until the Edge Function is deployed and verified, or the
> forms will stop working.

## 6. Email notifications (optional)

The `submit` Edge Function can email you when a contact message or testimonial
arrives, and send the visitor an auto-reply. It uses [Resend](https://resend.com)
(generous free tier). With no key set, this is a no-op — submissions still save.

1. Create a Resend account, **verify your sending domain**, and copy an API key.
2. Give the function its secrets and redeploy:
   ```bash
   supabase secrets set RESEND_API_KEY=re_xxx
   supabase secrets set CONTACT_FROM="Navya EdTech <hello@your-domain.com>"
   supabase secrets set CONTACT_NOTIFY_TO=team@your-domain.com   # comma-separated ok
   supabase functions deploy submit --no-verify-jwt
   ```
   `CONTACT_FROM` must use the domain you verified in Resend.

What gets sent:

| Trigger                | Email                                            |
| ---------------------- | ------------------------------------------------ |
| New contact message    | Team notification (Reply-To = visitor) + visitor auto-reply |
| New testimonial        | Team notification (pending review)               |

Emails are best-effort and run **after** the row is saved (off the response
path), so an email outage never fails a submission. Visitor text is HTML-escaped.

## 7. Per-IP rate limit

On top of the per-email DB throttle, the function rate-limits by client IP
(`register_ip_hit`, migration 0005) so a bot can't bypass the limit by rotating
the email field. The default is **8 submissions / 10 min per IP** across all
forms — tune `IP_MAX_HITS` / `IP_WINDOW` in
[`functions/submit/index.ts`](./functions/submit/index.ts). If the migration
isn't applied the function fails open (allows the request); the other layers
(Turnstile, per-email throttle) still apply.

## Security runbook

- **Anon key is public by design.** Every table is guarded by Row-Level Security
  (see `schema.sql`): anonymous visitors can only submit contact messages /
  unpublished testimonials and read published content. Never ship the
  **service_role** key to the browser.
- **Password-reset tokens.** The recovery token arrives in the URL and is
  consumed by `supabase-js` (`detectSessionInUrl: true`). The app never reads or
  logs it, and `src/lib/logger.ts` only logs explicit messages (no URLs/tokens).
  If you add logging around auth, do **not** log `window.location` or the
  session — keep tokens out of logs and out of Sentry.
- **Testimonial quotes are visitor-authored.** They're inserted unpublished and
  rendered as plain text only — never through Markdown / `dangerouslySetInnerHTML`
  (enforced by convention; see `src/services/testimonials.ts`).
- **Content Security Policy.** A CSP is set via `<meta>` in `index.html`. Headers
  that can't be delivered by meta (HSTS, `X-Frame-Options`,
  `X-Content-Type-Options`) require a CDN in front (e.g. Cloudflare) — GitHub
  Pages can't set them. If you add a third-party origin (analytics, CAPTCHA,
  media host), update the CSP's `connect-src` / `script-src` accordingly.
